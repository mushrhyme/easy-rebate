FIELD NAME IMMUTABILITY (CRITICAL)

All keys must match REFERENCE_JSON EXACTLY.

- mix languages
- create variations

STRUCTURE RULES (CRITICAL)

- Output must follow the structure of REFERENCE_JSON
- All keys and nested objects are mandatory unless explicitly removed by page_role rules

CRITICAL:
- Never collapse any object into a string
- Always preserve nested structure
- Missing values → null (not omission)


NATIVE JSON TYPE RULE (CRITICAL)

For every field in REFERENCE_JSON, preserve the same JSON data type.

- If a field is an object → it must remain an object
- If a field is an array → it must remain an array
- If a field is a scalar → it must remain a scalar

Never:
- serialize objects into strings
- serialize arrays into strings
- flatten nested objects into plain text

Examples:
Correct:
"recipient": { "company_name": "ABC" }

Wrong:
"recipient": "ABC"

Wrong:
"recipient": "{\"company_name\":\"ABC\"}"


TYPE ENFORCEMENT RULE (CRITICAL)

Do not change JSON data types under any circumstance.

If a field is:
- object → must remain object
- array → must remain array
- scalar → must remain scalar


GENERATION PROCEDURE (CRITICAL)

1. Determine page_role FIRST
2. Apply page-role-specific structure rules
3. Clone REFERENCE_JSON structure
4. Keep all keys identical
5. Extract values from TARGET_OCR
6. Infer タイプ


STRUCTURE EXCEPTION RULE (ITEMS) (CRITICAL)

items is controlled ONLY by page_role.

- items MUST exist
- items MUST be an array
- if no rows exist → "items": []

- omit the items key entirely
- do NOT output items: []
- do NOT output items: null
- do NOT inherit items from REFERENCE_JSON


PAGE ROLE RULES
