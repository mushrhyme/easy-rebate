MISSION

Your task is to extract TARGET_OCR table rows faithfully into one JSON object.

The output JSON must strictly follow the same structure and semantic system as REFERENCE_JSON.

REFERENCE_JSON is both:
- a structure reference
- a semantic labeling reference

Fields like タイプ should follow the same labeling logic shown in REFERENCE_JSON.

Classification fields such as タイプ and page_role:
- are NOT direct extraction fields
- may be inferred from document patterns and REFERENCE_JSON


IMPORTANT PRINCIPLES

All values must be extracted from TARGET_OCR,
EXCEPT for classification fields (タイプ, page_role).

タイプ is a classification field:
- it must NOT be extracted by literal matching only
- it must be inferred based on row patterns and REFERENCE_JSON

If a row matches the same semantic pattern as REFERENCE_JSON,
assign the same タイプ.

Do NOT return null for タイプ if a matching pattern clearly exists.


FIELD NAME IMMUTABILITY RULE (CRITICAL)

The JSON field names must be copied EXACTLY from REFERENCE_JSON.

Do NOT:
- invent new field names
- translate field names
- partially modify field names
- mix languages in field names
- create variations of existing field names

If a key does not exist in REFERENCE_JSON,
it must NEVER appear in the output.

FORBIDDEN mutations (examples):
- "受注先コード" must NEVER become "受注先_code", "受注先code", "受注先Code"
- "得意先コード" must NEVER become "得意先_code", "得意先code"
- Any Japanese key must NOT be partially converted to underscored or Latin form
- Do NOT add keys with underscores (_) unless they exist in REFERENCE_JSON

If you are uncertain about a key name, output null with the EXACT key from REFERENCE_JSON.
Never output both a correct and an incorrect variant of the same key.


STRUCTURE RULES

The output JSON schema must be a strict clone of REFERENCE_JSON keys,
EXCEPT for the items field.

This includes:
- object structure
- array structure
- key names
- key order

Only values may change.


STRUCTURE EXCEPTION RULE (ITEMS)

items is controlled ONLY by page_role.

If page_role = "detail":
- items MUST exist
- items MUST be an array
- if no rows exist → "items": []

If page_role = "cover", "summary", or "reply":
- omit the items key entirely
- do NOT output items: []
- do NOT inherit items from REFERENCE_JSON


GENERATION PROCEDURE

Follow strictly:

1. Copy the entire JSON structure from REFERENCE_JSON
2. Keep all keys exactly identical
3. Determine page_role
4. Apply items rule based on page_role
5. Fill values using TARGET_OCR
6. Infer タイプ using REFERENCE_JSON patterns


ROW FIDELITY RULE (HIGHEST PRIORITY)

Use TARGET_OCR as the ground truth for row count and row boundaries.

Rules:
- Keep each table row as one item.
- Do NOT merge rows.
- Do NOT split one row into multiple items unless the OCR clearly contains separate rows.
- Do NOT drop rows just because they differ from REFERENCE_JSON.
- REFERENCE_JSON is for key naming/format guidance, not for forcing item count.


PAGE ROLE RULES

page_role must be one of:

"cover"
"detail"
"summary"
"reply"

page_role is inferred from content (not explicitly written).

Priority:

1. Repeating item rows → "detail"
2. Totals / tax / summary → "summary"
3. Company info only → "cover"
4. Otherwise → "reply"


ROW EXTRACTION RULES

All recognizable rows must be extracted into items.

Do NOT:
- delete rows
- merge rows

Duplicates must be preserved.


VALUE EXTRACTION RULES

If value exists → extract  
If not → null

VALUE FORMAT ALIGNMENT RULE (HIGHEST PRIORITY)

For any field that exists in REFERENCE_JSON:

- The output format of the value MUST follow the format used in REFERENCE_JSON,
  not the raw format from TARGET_OCR.

This includes:
- presence or absence of unit symbols (e.g., 円, %)
- numeric formatting (decimal style, separators)
- text normalization style

Rules:
- If REFERENCE_JSON shows the value without a unit, remove the unit from TARGET_OCR value
- If REFERENCE_JSON includes a unit or symbol, preserve it accordingly
- The semantic value comes from TARGET_OCR, but the formatting comes from REFERENCE_JSON

REFERENCE_JSON defines the canonical representation format for each field.

REFERENCE_OCR COLUMN RULE

Use REFERENCE_OCR to understand column alignment.

In TARGET_OCR:
- extract by row alignment
- follow relative column order
- ignore spacing inconsistencies


AMOUNT NORMALIZATION RULE

"1:500" → "1500"
"12:600" → "12600"

Remove ":" only.
Do NOT invent digits.


OCR SPACE-SEPARATED NUMBER RULE

OCR often splits a single decimal number into space-separated groups.
Apply to numeric condition fields (条件, 単価, 販売価格, etc.):

Rule: Remove all spaces between digit groups, then treat the last 2 digits as decimal places.

Examples:
"71 00"   → "71.00"
"43 50"   → "43.50"
"1 42 00" → "142.00"
"80 00"   → "80.00"

Do NOT apply to: 商品コード, 請求番号, 得意先コード, 配送支店, or any ID/code field.
Do NOT apply if a decimal point already exists.


NUMBER RULE

Do NOT guess decimals.
Prefer raw OCR value if ambiguous.


備考 / 条件備考 RULES

Both must be populated if both exist.

備考:
- extract from 備考 column
- replace line breaks with space
- no tabs

条件備考:
- extract based on REFERENCE_OCR column
- normalize all whitespace → single space
- no tabs


DITTO RULE

Symbols:
〃 ″ " ' // ／ 1/ /

If cell contains only these → null

Never output these symbols.


FAIL-SAFE RULE

If uncertain:
- keep structure
- return null for values


JSON OUTPUT RULES

Return exactly ONE JSON object

Do NOT include:
- explanations
- comments
- markdown
- extra text


FINAL OUTPUT

Return exactly one valid JSON object
