{example_ocr}

REFERENCE_JSON (정답 구조 예시):
{example_answer_str}

TARGET_OCR (실제 처리 대상 OCR 텍스트):
{ocr_text}


MISSION
너의 작업은 TARGET_OCR 문서를 **완전히 재구조화하여 JSON으로 변환하는 것**이다.

출력 JSON은 반드시 **REFERENCE_JSON과 동일한 구조와 의미 체계**를 따라야 한다.

중요 원칙:

1. 모든 데이터는 반드시 **TARGET_OCR에서만 추출**한다.
2. 문서에 없는 값은 절대 생성하거나 추측하지 않는다.
3. JSON **필드명(키)은 REFERENCE_JSON과 완전히 동일하게 유지**한다.
4. 필드명을 번역하거나 수정하거나 한글을 섞으면 안 된다.
5. 출력은 반드시 **유효한 JSON 하나만 반환**해야 한다.


OUTPUT STRUCTURE RULES

1. 출력 JSON의 **전체 구조는 REFERENCE_JSON과 동일해야 한다.**

2. 최상위 구조
   - 객체 / 배열 구조
   - 키 이름
   - 키 순서

   모두 REFERENCE_JSON과 동일하게 유지한다.

3. JSON 출력 규칙

- 반드시 하나의 JSON 객체만 출력한다
- 설명문, 주석, 코드블록을 포함하지 않는다
- JSON 앞뒤에 어떤 텍스트도 추가하지 않는다


FIELD NAME RULES (최우선 규칙)

모든 JSON 필드명은 **REFERENCE_JSON과 정확히 동일해야 한다.**

절대 금지:

- 필드명 번역
- 한글 혼합
- 철자 변경

예:

허용
"請求伝票番号"

금지
"請求伝票번호"
"청구전표번호"


PAGE ROLE RULES

page_role은 반드시 문자열이며 아래 값 중 하나여야 한다.

"cover"
"detail"
"summary"
"reply"

page_role 판단 규칙 (우선순위)

1. 상품/조건/수량/금액 등 **반복 행 구조가 존재하면 → detail**

2. 합계 / 세금 / 총 청구금액 등  
   **요약 금액 정보 중심이면 → summary**

3. 회사명 / 주소 / 청구서 제목 / 문서 식별 정보 중심이고  
   상품 반복 행이 없으며  
   문서의 첫 페이지이면 → cover

4. cover와 유사하지만  
   문서의 첫 페이지가 아니면 → reply


ITEMS ARRAY RULES

detail 페이지에서:

items는 **항상 배열([])** 이어야 한다.

- 상품 행이 존재하면 → items 배열에 모두 포함
- 상품 행이 없으면 → 빈 배열 []

절대 금지

items: null


cover / summary / reply 페이지

items 키 자체를 생성하지 않는다.


ROW EXTRACTION RULES

OCR에서 식별 가능한 모든 상품 행을 items 배열로 추출한다.

중요:

- 동일한 행이 반복되더라도 삭제하거나 병합하지 않는다
- OCR에 존재하는 상품 행은 모두 유지한다


VALUE EXTRACTION RULES

모든 필드는 다음 규칙을 따른다.

값이 문서에 존재하면 → 해당 값을 추출  
값이 문서에 없으면 → null 반환


AMOUNT NORMALIZATION RULE

금액 필드 (金額 / 請求金額 등)

OCR이 회계 점선을 콜론(:)으로 인식하는 경우가 있다.

예

1:500
1:566
12:600

이 경우

콜론을 제거하고 숫자만 이어 붙인다.

예

"1:500" → "1500"
"12:600" → "12600"


NUMBER FORMAT NORMALIZATION

단가 / 조건 / 조건％ / 판매가격 등 숫자 필드

OCR에서 다음 형태가 나타날 수 있다.

78 00
56 00
126 00

의미는

7800
5600
12600

하지만 출력 형식은 반드시 **소수점 표기**로 통일한다.

예

"78 00" → "78.00"
"56 00" → "56.00"
"126 00" → "126.00"


備考 / 条件備考 RULES

備考와条件備考는 서로 다른 필드이다.

REFERENCE_JSON에 두 키가 존재하면  
각 상품 행마다 반드시 값을 채운다.


備考

표에 "備考" 헤더가 있는 열이다.

REFERENCE_OCR에서 備考가 위치한 **같은 상대 위치의 셀**을 TARGET_OCR에서 찾아 추출한다.

특징

- 담당자명
- 짧은 텍스트

예

栗田 康広


출력 규칙

줄바꿈(\n)은 공백 하나로 변환한다.  
탭은 사용하지 않는다.


条件備考

REFERENCE_JSON에 조건備考 키가 존재하지만  
헤더에 "条件備考"가 없는 경우가 있다.

이 경우 다음 규칙을 사용한다.

1. REFERENCE_OCR에서 条件備考 값이 위치한 **열 위치**를 확인한다

2. TARGET_OCR에서 동일한 상대 위치의 텍스트를 찾는다

3. 일반적으로 상품 행 **오른쪽 끝에 위치한 긴 텍스트 블록**이다


예

ミセダシ ウリアゲ キホンジョウケン

ＳＣ：７桁コード
Ｎ１ Ｋ


출력 규칙

조건備考 문자열 내부에서

- 연속 공백
- 탭
- 전각 공백

모두 **하나의 일반 공백**으로 정규화한다.

JSON 문자열 안에는 **탭을 사용하지 않는다.**


SAME-AS SYMBOL RULE (同上 처리)

표에서 다음 기호는  
"위 행과 동일"을 의미한다.

예

〃
//
1/
″


OCR에서 셀이 이러한 기호만 포함하는 경우

해당 필드는 **null**로 반환한다.

예

商品名 = null

후처리에서 이전 행 값으로 채워진다.


IMPORTANT

이 작업은 질문에 대한 부분 응답이 아니라  
OCR 문서 전체를 **정규화된 JSON 구조로 변환하는 작업**이다.

TARGET_OCR은 출력 범위를 제한하기 위한 것이 아니라  
문서 구조를 분석하기 위한 입력 데이터이다.


FINAL OUTPUT

반드시 **유효한 JSON 하나만 출력한다.**
