# 검토 및 정답 생성 로직

## 1. 화면 구분 (혼동 방지)

### 1.1 검토 탭 = 検索 탭 (CustomerSearch)

문서를 검토하는 메인 화면이다.

**기능**
- 문서 목록 조회
- 문서 선택
- 페이지 이동
- 페이지 내용 확인
- 페이지 편집
- 저장
- 재분석
- 학습 요청

**중요 규칙**
- **페이지 이동은 절대 분석을 자동 실행하지 않는다.**
- **분석은 버튼으로만 실행된다.**

### 1.2 解答作成 탭 (AnswerKeyTab)

검토 탭에서 현재 페이지 1장만 브릿지하여 편집하는 화면이다.

**특징**
- 단일 페이지 전용
- 페이지 이동 기능 없음
- 자동 분석 없음

**가능한 작업**
- 편집
- 저장
- 재분석
- 학습 요청

**정리**
- **검토 탭** → 문서 검토
- **解答作成 탭** → 단일 페이지 편집

---

## 2. 전체 프로세스 개요

```
PDF 업로드
    ↓
이미지 생성 + 전체 페이지 OCR·분석 (페이지 완료 시마다 DB 저장)
    ↓
form_type 확정 (분석 결과 기준)
    ↓
검토 탭에서 문서 검토
    ↓
사용자 수정
    ↓
저장
    ↓
필요 시 재분석
    ↓
필요 시 학습 요청
    ↓
검토 완료
    ↓
엑셀 생성 및 다운로드
```

---

## 3. 문서 단위 vs 페이지 단위 데이터

### 3.1 문서 단위 데이터

문서 전체에 적용되는 속성:
- `form_type`
- `upload_channel`
- `data_year`
- `data_month`
- `total_pages`
- `current_vector_version`

**설명**
- `form_type`은 업로드 시 분석 결과로 결정. 이후 자동 변경 없음.
- `form_type`이 잘못된 경우: 페이지 수정 → 학습 요청 → 이후 분석 결과 개선.

### 3.2 페이지 단위 데이터

각 페이지는 독립적인 분석 상태를 가진다.

**예**
- `page_no`
- `ocr_text`
- `analysis_result`
- `analyzed_vector_version`
- `last_analyzed_at`

**설명**
- `analyzed_vector_version`: 이 페이지가 어떤 벡터 DB 버전으로 분석되었는지 기록.

## 4. 업로드 탭 동작

PDF 업로드 시 수행되는 작업 (현재 구현 기준):

### 4.1 이미지 생성
- PDF → 페이지 이미지 생성 후 DB·저장소에 저장.

### 4.2 OCR 및 분석
- **현재**: 업로드 시 **전체 페이지**에 대해 OCR + RAG + LLM 분석을 수행.
- 페이지별 완료 시마다 해당 페이지 데이터를 DB에 저장 (`page_data_current.ocr_text`, `page_meta`, `items` 등).

**중요 규칙**
- **재분석** 시에는 OCR을 **DB에서만** 읽고, OCR 재실행은 하지 않음 (단일 페이지 분석 API 등).

### 4.3 form_type 결정
- 분석 결과(페이지별 `_rag_reference` 등)를 바탕으로 `documents_current.form_type` 설정.

---

## 5. 페이지 이동 규칙

**페이지 이동은 단순 조회 기능이다.**

**중요 규칙**
- **페이지 이동은 분석을 실행하지 않는다.**

**페이지 이동 시 수행되는 작업**
- OCR 표시
- 기존 분석 결과 표시
- 분석이 필요하면 사용자가 **버튼**을 눌러 실행한다.

---

## 6. 재분석 정책

**분석은 자동 실행되지 않는다.** 분석 실행 방법은 **버튼 2개만** 존재한다.

### 6.1 현재 페이지 재분석
- **버튼**: このページ再分析
- **동작**: 현재 페이지만 RAG + LLM 재실행. OCR은 DB 사용.

### 6.2 현재 페이지 이후 전체 재분석
- **버튼**: このページ以降の全ページを再分析
- **동작**: 현재 페이지 포함, 마지막 페이지까지 재분석. **병렬 처리**로 실행하여 대기 시간 단축.
- **병렬**: 동시에 여러 페이지 분석(동시도 상한 예: 3~5). API rate limit·리소스 고려.
- **예**: 4페이지에서 실행 → 4 ~ 마지막 페이지를 병렬로 재분석.

### 6.3 재분석 진행 중 학습 요청
- **규칙**: 재분석(이 페이지 이후 전체)이 진행 중일 때 사용자가 학습 요청을 하면, **진행 중인 재분석을 중단**한다.
- 이유: 학습 요청 시 벡터 버전이 올라가므로, 이전 버전으로 돌고 있던 재분석 결과는 무의미해짐. 필요 시 사용자가 중단 후 다시 재분석 버튼을 실행.

---

## 7. 학습 요청

- **버튼**: 学習リクエスト
- **동작**: 현재 페이지 데이터를 벡터 DB 반영 → `current_vector_version` +1.
- **예**: v1 → v2 → v3

### 7.1 학습 요청 조건
- 편집 중이면 요청 불가.
- 저장되지 않은 수정 존재 → "저장 후 학습 요청해주세요" 알림.
- 해당 문서에 **재분석(이 페이지 이후 전체)이 진행 중**이면 → 진행 중인 재분석 중단 후 학습 반영 (6.3 참고).

---

## 8. 검토 탭 기능

검토 탭에서 제공하는 버튼:
- **保存**
- **このページ再分析**
- **このページ以降の全ページを再分析**
- **学習リクエスト**
- **解答作成**

### 8.1 저장
- **버튼**: 保存 / Ctrl + S
- **현재**: 검토 탭에서는 **편집 중인 행(item)만** 행 단위 PATCH로 저장. 자동 저장 없음.
- 解答作成 탭에서는 해당 페이지만 PUT answer-json으로 저장.

### 8.2 解答作成
- **버튼**: 解答作成
- 현재 페이지 1장만 AnswerKeyTab으로 전달.

---

## 9. 解答作成 탭 동작

- 검토 탭에서 전달된 **단일 페이지만** 표시.

**가능 작업**
- 편집
- 저장
- 재분석
- 학습 요청

**제한**
- 페이지 이동 없음
- 자동 분석 없음

**버튼**
- **検討タブに復帰**: 검토 탭으로 복귀.

---

## 10. 18페이지 문서 예시

1. **업로드 (현재 구현)**
   - 전체 페이지(1~18)에 대해 OCR·분석 수행, 페이지 완료 시마다 DB 저장.
   - form_type 확정. 학습 요청 전까지 `current_vector_version` = 1 등.

2. **검토 진행**
   - 1페이지 확인 → 2페이지 확인 → 3페이지 확인
   - **자동 분석 없음.**

3. **3페이지 수정 후 학습 요청**
   - `vector_version` = v2

4. **4페이지 이후 반영 필요 시**
   - 「このページ以降の全ページを再分析」 실행
   - 재분석 범위: **4 ~ 18**을 **병렬**로 재분석.

## 11. 문서 검토 완료 조건

엑셀 생성 전 확인:
- 저장되지 않은 수정 없음
- 필요한 페이지 재분석 완료

---

## 12. 핵심 원칙

| 항목 | 규칙 |
|------|------|
| **OCR** | 업로드 시 페이지별 OCR·분석 후 DB 저장. **재분석** 시에는 OCR은 DB에서만 읽음, 재실행 없음. |
| **분석** | 페이지 이동 시 자동 분석 없음. |
| **분석 실행** | 버튼으로만 실행 (이 페이지 재분석 / 이 페이지 이후 전체 재분석). |
| **분석 범위** | 현재 페이지 **또는** 현재 페이지 이후 전체. |

## 13. 구현 필수 데이터 (init_database.sql 반영)

**documents_current**
- `form_type`, `total_pages`, `current_vector_version`

**page_data_current**
- `page_number`, `page_role`, `page_meta`
- `ocr_text`, `analyzed_vector_version`, `last_analyzed_at`
- `last_edited_at`, `last_edited_by_user_id`, `is_rag_candidate`

(분석 결과·메타는 `page_meta` JSONB 및 items_current 등에 저장됨.)
